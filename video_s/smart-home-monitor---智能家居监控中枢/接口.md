# 浏览器前端 ↔ 中枢 HTTP 约定

下文为 **浏览器与告警中枢** 之间的交互方式与数据格式：**仅用 HTTP**，无 WebSocket、无 MQTT。`POST` 请求统一携带请求头 **`Content-Type: application/json`**，请求体为 JSON 对象。

**Base URL**：由部署决定。开发时常见做法是用开发服务器把以 **`/api`** 开头的路径转发到中枢；生产构建时前端会把完整 API 根地址拼成 **`https://{页面主机名}:8443`**（可在前端配置里修改端口或规则）。以下路径均相对于该 Base URL。

---

## 1. 轮询（读侧推荐用法）

前端以固定间隔（示例工程默认 **1000 ms**）**并行**发起：

1. **`GET /api/alerts?limit=…`**
2. **`GET /api/telemetry`**

- 若告警请求成功，界面可视为与中枢连接正常；失败则视为断开。
- 告警列表为 **JSON 数组**；前端若只关心最新一条，常取 **`limit=1`**。
- `limit`：整数，**1～200**，缺省若未约定则由中枢自定（实现可默认 20）。

---

## 2. `GET /api/alerts`

### 查询参数

| 参数 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `limit` | 整数 | 1～200 | 返回条数上限 |

### 响应

**200**：`application/json`，**对象数组**，每条为一条告警。元素字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `device_id` | 字符串 | 设备标识 |
| `event_type` | 字符串 | 事件类型 |
| `confidence` | 数值 | 置信度 |
| `source_timestamp` | 字符串或 `null` | 设备侧时间（可选） |
| `received_at` | 字符串 | 中枢收到时间（常用 ISO8601 且带 `Z`） |
| `level` | 字符串 | 如 `critical`、`warning`、`info` |
| `message` | 字符串 | 摘要说明 |

**数组顺序**：建议 **下标 0 为最新**（便于前端直接取 `[0]`）。

### 响应示例（界面红警涉及的三种 `event_type` 各一条）

数组仅为示例；真实接口在同一路径下仍返回 **`AlertOut` 对象数组**，顺序以中枢为准（建议最新在下标 0）。

```json
[
  {
    "device_id": "sensor-fire-01",
    "event_type": "fire_alarm",
    "confidence": 0.91,
    "source_timestamp": "2026-05-05T12:03:00Z",
    "received_at": "2026-05-05T12:03:00.512000Z",
    "level": "critical",
    "message": "Fire or smoke alarm"
  },
  {
    "device_id": "sensor-smoke-01",
    "event_type": "smoke_alarm",
    "confidence": 0.87,
    "source_timestamp": "2026-05-05T12:02:00Z",
    "received_at": "2026-05-05T12:02:15.120000Z",
    "level": "critical",
    "message": "Fire or smoke alarm"
  },
  {
    "device_id": "camera-fall-01",
    "event_type": "fall_detected",
    "confidence": 0.92,
    "source_timestamp": "2026-05-05T12:01:00Z",
    "received_at": "2026-05-05T12:01:08.330000Z",
    "level": "warning",
    "message": "Fall detected"
  }
]
```

**三种类型对照（前端据此之一即可进红警 UI）：**

| `event_type` | 典型 `level` | 典型 `message`（中枢可按规则生成） |
|--------------|--------------|-------------------------------------|
| `fire_alarm` | `critical` | 如：`Fire or smoke alarm` |
| `smoke_alarm` | `critical` | 同上（火焰/烟雾在服务端常归为同类文案） |
| `fall_detected` | `warning` 或 `critical` | 置信度较低时常为 `Fall detected`；若规则判定为高置信跌倒，可为 `critical` 且 message 如 `Fall detected with very high confidence` |

空列表 `[]` 表示当前缓冲区内无告警。

### 与界面告警态的约定（前端逻辑）

仅当 `event_type`（**不区分大小写**）为以下之一时，进入红色告警 UI：

- `fire_alarm`
- `smoke_alarm`
- `fall_detected`

去重：可用 `device_id`、`event_type`、`confidence`、`received_at`（或等价指纹）避免同一条反复触发。

---

## 3. `GET /api/telemetry`

### 响应

**200**：`application/json`

```json
{ "temperature_celsius": 24.5 }
```

若尚无温度数据：

```json
{ "temperature_celsius": null }
```

前端在无有效数值时可自行默认显示（例如 24°C），属 UI 策略，非协议硬性规定。

---

## 4. `GET /api/status`（可选）

### 响应示例

**200**：`application/json`

```json
{
  "status": "running",
  "service": "aiot-alert-listener",
  "ingest": {
    "http_alert": true,
    "http_sensor": true
  },
  "buffer_size": 10
}
```

字段含义以实现为准；浏览器可用于探活。

---

## 5. `POST /api/ptz`

云台移动指令。

### 请求体

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `message_type` | 字符串 | 固定为 **`移动`** | 与当前前端发送一致 |
| `direction` | 整数 | **1～4** | 1＝上，2＝下，3＝左，4＝右 |

### 请求示例

```json
{
  "message_type": "移动",
  "direction": 1
}
```

### 响应示例

**200**：`application/json`（形状以实现为准，常见如下）

```json
{
  "ok": true,
  "received": {
    "message_type": "移动",
    "direction": 1
  }
}
```

---

## 6. `POST /api/mode`

界面模式同步。请求体**只有一个字段** `mode`，三种取值对应前端三种模式切换：

| `mode` 取值 | 含义（与界面一致） |
|-------------|-------------------|
| **`monitor`** | 监控 |
| **`search`** | 查找 |
| **`track`** | 追踪 |

### 请求体

| 字段 | 类型 | 约束 |
|------|------|------|
| `mode` | 字符串 | 必须是上表中三者之一；建议全小写；若中枢校验，通常归一化为小写再比对 |

### 请求示例（三种各一种）

```json
{ "mode": "monitor" }
```

```json
{ "mode": "search" }
```

```json
{ "mode": "track" }
```

### 响应示例

```json
{
  "ok": true,
  "received": { "mode": "search" }
}
```

（`received.mode` 与请求体一致即可，以上以查找模式为例。）

---

## 7. `POST /api/search`

查找关键字提交。

### 请求体

| 字段 | 类型 | 说明 |
|------|------|------|
| `query` | 字符串 | 可为空字符串 |

### 请求示例

```json
{ "query": "关键字" }
```

### 响应示例

```json
{
  "ok": true,
  "received": { "query": "关键字" }
}
```

---

## 8. `POST /api/track/refresh`

刷新追踪状态。

### 请求体

| 字段 | 类型 | 约束 |
|------|------|------|
| `action` | 字符串 | 固定为 **`refresh_track`** |

### 请求示例

```json
{ "action": "refresh_track" }
```

### 响应示例

```json
{
  "ok": true,
  "received": { "action": "refresh_track" }
}
```

---

## 9. 跨域

若页面 URL 与中枢 API **不同源**，浏览器会执行 CORS 校验；中枢需在响应中返回与前端来源匹配的 **`Access-Control-Allow-Origin`** 等头，否则 `fetch` 会被浏览器拦截。具体允许列表由部署决定。

---

## 10. 方法汇总

| 方法 | 路径 | 作用 |
|------|------|------|
| GET | `/api/alerts?limit=n` | 拉取告警列表 |
| GET | `/api/telemetry` | 拉取环境温度 |
| GET | `/api/status` | 可选，服务状态 |
| POST | `/api/ptz` | 云台 |
| POST | `/api/mode` | 模式 |
| POST | `/api/search` | 查找 |
| POST | `/api/track/refresh` | 刷新追踪 |

以上为前端与中枢交互所需的 **路径、参数、请求与响应 JSON 形状及取值约束**。
